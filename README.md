# MathClass — полное описание сайта

## 1) Что это за сайт
**MathClass** — это одностраничный web-проект на чистом HTML/CSS/JS с авторизацией через Firebase и ролевой моделью доступа.

Проект реализует:
- регистрацию и вход по email/password;
- проверку безопасности по IP;
- верификацию профиля через отправку имени;
- разделение прав по ролям (`user`, `moderator`, `elder`);
- инструменты модерации и журнал действий.

---

## 2) Технологии
- **Frontend:** HTML + CSS + vanilla JavaScript (ES modules).
- **Backend-as-a-Service:** Firebase Authentication + Cloud Firestore.
- **Клиентские библиотеки:** Firebase JS SDK `10.12.5` (через CDN).

Конфигурация Firebase подключена в `firebase.js`.

---

## 3) Структура страниц

### `index.html` — вход/регистрация
- Поля email + password.
- Кнопки «Регистрация» и «Вход».
- Отдельный экран вечной блокировки («Доступ навсегда закрыт»), если IP/аккаунт заблокирован.

### `name.html` — анкета имени
- Доступна только не верифицированным пользователям.
- Пользователь отправляет имя на проверку.
- После отправки форма блокируется, пока модератор/elder не включит верификацию.

### `main.html` — главная пользователя
- Показывает приветствие.
- Для не верифицированных показывает блок «Нет верификации».
- Есть проверка перехода в модераторские разделы.

### `moderator.html` — панель модератора
- Две вкладки: верифицированные и не верифицированные пользователи.
- Модальное окно профиля игрока.
- Действие: поставить/снять верификацию.

### `elder.html` — панель elder (расширенный контроль)
- Всё, что есть у модератора.
- Дополнительные действия: смена роли, вечный бан аккаунта, бан IP, поиск связанных аккаунтов по IP, массовый бан по IP.
- Метрики (всего аккаунтов, заблокировано, подозрительные IP).
- Архив действий (`audit_logs`) с фильтрами.
- Отдельное модальное «Меню пресечения нарушений».

---

## 4) Роли и права

### `user`
- Может зарегистрироваться и войти.
- До верификации работает только с формой имени.
- После верификации получает доступ к `main.html`.

### `moderator`
- Видит список пользователей.
- Может включать/снимать верификацию.
- Не может блокировать IP/аккаунты навсегда и не видит архив elder-уровня.

### `elder`
- Имеет полный доступ к модерации.
- Может назначать роли (`user`, `moderator`, `elder`).
- Может делать пожизненные блокировки аккаунтов и IP.
- Имеет доступ к журналу действий.

---

## 5) Логика доступа
Центральная проверка прав реализована в `auth-guard.js`:

- Для всех страниц проверяется IP (через `api.ipify.org`).
- Если IP не определился — fail-closed (доступ запрещается).
- Если IP есть в `blocked_ips` и помечен `blocked: true` — доступ запрещается.
- Для авторизованных пользователей читается `users/{uid}`:
  - `blockedForever: true` → полный запрет доступа;
  - `verified` и `role` проверяются по политике конкретной страницы.

Политики страниц:
- `index` — страница входа/регистрации;
- `main` — требует верификацию;
- `name` — только для не верифицированных;
- `moderator` — только `moderator` и `elder`, плюс верификация;
- `elder` — только `elder`, плюс верификация.

## 6) Данные Firestore (модель)

### Коллекция `users`
Документ: `users/{uid}`

Ключевые поля:
- `email: string`
- `name: string | null`
- `nameSubmitted: boolean`
- `verified: boolean`
- `role: 'user' | 'moderator' | 'elder'`
- `blockedForever: boolean`
- `createdIp: string`
- `lastIp: string`
- `createdAt`, `updatedAt`, `verifiedAt`, `blockedAt`, `submittedAt`: ISO timestamp
- `blockedBy: uid` (кто заблокировал)

### Коллекция `blocked_ips`
Документ: `blocked_ips/{ip}`

Ключевые поля:
- `blocked: boolean`
- `reason: string` (например `forever`)
- `blockedAt: ISO timestamp`
- `blockedBy: uid`

### Коллекция `audit_logs`
Документ: `audit_logs/{autoId}`

Ключевые поля:
- `action: string` (например `verify_user`, `block_ip_forever`)
- `actorId`, `actorEmail`
- `targetId`, `targetEmail`, `targetIp`
- `details: string | null`
- `createdAt: ISO timestamp`

## 7) Аудит и безопасность
Сайт пишет события в `audit_logs` почти для всех критичных действий:
- успешный/неуспешный вход и регистрация;
- блокировки (аккаунт/IP);
- смена роли;
- операции верификации.

Подход к безопасности:
- **Fail-closed** при проблемах с определением IP/чтением критичных данных.
- Вечный бан может включаться и по аккаунту, и по IP.
- Основная авторизация/ограничения дублируются Firestore Rules.

## 8) Пользовательские сценарии

### Сценарий A: новый пользователь
1. Регистрируется на `index.html`.
2. Попадает на `name.html`.
3. Отправляет имя, ждёт проверки.
4. После верификации модератором получает доступ на `main.html`.

### Сценарий B: модерация
1. Пользователь с ролью `moderator` заходит в `moderator.html`.
2. Открывает карточку игрока.
3. Включает/выключает верификацию.

### Сценарий C: пресечение нарушений (elder)
1. Elder открывает `elder.html`.
2. Ищет подозрительные связки по IP.
3. Блокирует IP и/или массово блокирует связанные аккаунты.
4. Проверяет историю действий в журнале.

---

## 9) Локальный запуск
Проект статический; нужен любой HTTP-сервер (чтобы работали ES-модули).

Примеры:
- Python: `python3 -m http.server 8000`
- Node: `npx serve .`

Далее открыть `http://localhost:8000/index.html`.

---

## 10) Что важно настроить перед продом
1. Обновить Firestore Rules (см. `FIRESTORE_RULES.md`).
2. Проверить назначение роли `elder` хотя бы одному доверенному аккаунту.
3. Ограничить и защитить Firebase API-ключ через домены/настройки проекта.
4. Добавить Cloud Functions/серверную валидацию для критичных операций (по желанию, для усиления безопасности).
5. Добавить экспорт и резервное копирование `audit_logs`.

---

## 11) Кратко о файлах
- `index.js` — auth flow + создание/обновление профиля + аудит входа.
- `name.js` — отправка имени и ожидание верификации.
- `main.js` — главная и переходы по роли.
- `moderator.js` — базовая модерация (верификация).
- `elder.js` — расширенная админ-панель и контроль нарушений.
- `auth-guard.js` — единый guard доступа.
- `firebase.js` — инициализация SDK и экспорт методов.
- `styles.css` — общие стили интерфейса.
